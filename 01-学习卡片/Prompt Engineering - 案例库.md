# Prompt Engineering：案例库

这份笔记只做一件事。

不讲大词。
不讲历史。
只看案例。

每个案例都回答 3 个问题：

1. 差 Prompt 差在哪
2. 好 Prompt 好在哪
3. 我下次怎么复用

## 案例 1：总结岗位描述

### 差 Prompt

```text
帮我总结这个岗位需要什么能力。
```

### 好 Prompt

```text
你现在扮演岗位分析师。

任务：
根据下面的岗位描述，提炼 5 个最核心能力，并按 P0 / P1 / P2 排优先级。

输出要求：
1. 用中文
2. 输出表格，字段为：能力 / 为什么重要 / 优先级
3. 不要照抄原文
4. 不要超过 200 字

材料：
[岗位描述]
```

### 差别

差 Prompt 最大的问题，是它把“总结”说了，却没把“怎么总结”说清楚。

好 Prompt 把任务、格式、长度和边界一次讲明白了。

### 记住

不是让模型“帮我想想”。

而是让模型“按这个方式给我结果”。

## 案例 2：把会议记录整理成 PRD

### 差 Prompt

```text
把这段会议记录整理一下。
```

### 好 Prompt

```text
你现在扮演产品经理助理。

任务：
把下面的会议记录整理成一版 PRD 草稿。

输出结构固定为：
1. 背景
2. 目标
3. 用户问题
4. 功能方案
5. 待确认问题

要求：
- 用中文
- 不要补不存在的信息
- 不确定的地方标记“待确认”
```

### 差别

“整理一下”太空了。

模型不知道你要摘要、纪要，还是 PRD。

好 Prompt 直接规定了目标文档是什么。

### 记住

如果你知道最后想要什么文档，就直接说。

不要让模型猜。

## 案例 3：让 AI 帮你写学习计划

### 差 Prompt

```text
帮我做一个学习计划。
```

### 好 Prompt

```text
你现在扮演 AI 产品学习教练。

任务：
根据我的目标，给我一个 7 天冲刺计划。

要求：
1. 每天只保留 1 个主目标
2. 每天都要有一个可见产物
3. 输出格式固定为：Day / 任务 / 产物 / 判断标准
4. 不要写空话，不要写长期愿景

背景：
[我的目标和当前能力]
```

### 差别

差 Prompt 会得到一份看起来很完整、其实做不完的计划。

好 Prompt 先限制了节奏和产物。

### 记住

计划如果不能执行，就只是更长的安慰。

## 案例 4：让 AI 改写一段表达

### 差 Prompt

```text
帮我润色一下这段话。
```

### 好 Prompt

```text
任务：
把下面这段话改写得更简洁、更干净、更像口语表达。

要求：
1. 保留原意
2. 删掉空话和重复话
3. 句子尽量短
4. 不要写成营销文案
5. 控制在原文长度的 70% 左右

原文：
[文本]
```

### 差别

“润色”这个词太模糊。

有人理解成更华丽。
有人理解成更正式。

但你真正想要的可能只是更清楚。

### 记住

少用抽象词。

多说你想让它“变成什么样”。

## 案例 5：让 AI 帮你写一个 skill 草稿

### 差 Prompt

```text
帮我做一个 skill。
```

### 好 Prompt

```text
你现在扮演 AI 工作流设计助手。

任务：
根据下面的重复任务，写一版 skill 草稿。

这个 skill 要解决：
[任务]

输出要求：
1. 先写 skill 目标
2. 再写输入是什么
3. 再写输出是什么
4. 再写执行步骤
5. 再写什么情况不该用这个 skill

要求：
- 用中文
- 不要写空泛原则
- 要像真实可执行的说明
```

### 差别

“做一个 skill”听起来像任务。

其实它什么都没说。

skill 最关键的是边界。

### 记住

越是要复用的东西，越要先讲清楚：

- 它解决什么
- 它不解决什么

## 最后一句

好 Prompt 不是更会说话。

好 Prompt 是更会定义任务。
