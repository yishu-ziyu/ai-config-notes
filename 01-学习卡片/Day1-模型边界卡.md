# Day1 模型边界卡

## 任务背景

在同一模型、同一输入下，对比 3 种 Prompt 写法对“岗位能力清单去重与优先级提炼”任务的影响。

## 本次实验暴露出的模型边界

### 1. 泛化总结默认会滑向“教学模式”

当 Prompt 只要求“提炼重点 + 给建议”时，模型更容易输出长篇解释和学习路径，而不是可直接沉淀的知识库结论。

### 2. 结构化约束能明显提升可用性

明确要求：

- 模块数量
- 固定输出结构
- 排除项
- 优先级

会显著提升结果的聚焦程度。

### 3. 输出预算会直接影响实验结论

这次 3 个版本都触发了 `max_tokens` 截断，说明：

- 当前模型或兼容接口存在隐藏思考开销
- 即使用户要求“简洁”，也不代表一定不会超预算
- Prompt 实验不能只看质量，还要看预算控制

### 4. 单次运行不能证明稳定性

当前只能得出“单次观察下 C 最好”，还不能得出“C 稳定最好”。

真正要判断稳定性，至少需要：

1. 同 Prompt 多次运行
2. 不同输入样本重复测试
3. 明确通过标准

## 对 AI 产品经理的启发

1. Prompt 设计不是写得漂亮，而是要服务于任务目标。
2. 好 Prompt 不只是提升效果，也是在控制成本和输出格式。
3. 如果任务目标是“沉淀产品资产”，Prompt 必须显式约束结构和边界。

## 当前结论

对这类“长文本去重 + 优先级判断 + 输出结论”的任务，强结构化 Prompt 明显优于泛化总结型 Prompt。
